# ADR-019: Portfolio-Level Experiment Program Optimization

- **Status**: Proposed
- **Date**: 2026-03-19
- **Author**: Agent-5 (Management) / Platform Architecture
- **Supersedes**: None

## Context

Kaizen optimizes each experiment independently: sample size is planned per-experiment, metrics are analyzed per-experiment, and ship/no-ship decisions are made per-experiment. There is no mechanism that optimizes the *portfolio* of experiments as a whole — the aggregate business value generated by the entire experimentation program across dozens of concurrent and sequential experiments.

Two Netflix papers (2024–2025) reframe experimentation as a resource allocation problem:

1. **"Optimizing Returns from Experimentation Programs" (EC '25)**: Derives optimal policies for allocating limited user populations across concurrent experiments. The counterintuitive finding: when implementation costs are negligible (true for recommendation changes), firms should run **more, lower-powered tests with relaxed p-value thresholds** to maximize cumulative business returns. The optimal alpha is often 0.10–0.20, not the traditional 0.05.

2. **"Evaluating Decision Rules Across Many Weak Experiments" (KDD '25, Best Paper Award in Applied Data Science)**: Proposes data-splitting to avoid winner's curse bias when evaluating decision rules (including alpha thresholds and metric selection rules) across a portfolio of past experiments. Netflix uses this to select proxy metrics and calibrate decision thresholds for all personalization A/B tests.

Netflix also developed **automated annualized impact projection** — translating 2–4 week experiment results into annualized revenue impact using surrogate index approaches combined with transportability assumptions across monthly cohorts. Their **Heterogeneous Treatment Effects infrastructure** (TechBlog, November 2025) integrates CATE analysis into every experimentation phase: discovery, hypothesis generation, ship/no-ship decisions, and annualized impact estimation.

Spotify's **Experiments with Learning (EwL) framework** (September 2025) provides a complementary meta-metric: classifying every experiment as "learning" or "no learning." Spotify's learning rate is ~64% vs. ~12% win rate, demonstrating that most experiment value comes from detecting regressions and confirming neutrality, not from finding winners.

## Decision

Add portfolio-level analytics and optimization to M5 (Management) and M6 (UI) that treat the experimentation program as a resource allocation problem.

### Component 1: Experiment Portfolio Dashboard

New M6 page (`/portfolio`) showing aggregate experimentation program metrics:

| Metric | Description | Computation |
|--------|-------------|-------------|
| **Win rate** | % of concluded experiments with significant positive primary metric | Count(significant & positive) / Count(concluded) |
| **Learning rate** (EwL) | % of concluded experiments that produced an actionable decision (ship, revert, or informed pivot) | Requires experimenter annotation at conclusion time |
| **Annualized impact** | Projected total revenue/retention impact of all shipped experiments | Sum of per-experiment annualized projections |
| **Traffic utilization** | % of total user population currently in at least one experiment | Union of all active experiment allocations |
| **Experiment throughput** | Experiments concluded per week/month | Time series of CONCLUDED transitions |
| **Average experiment duration** | Median days from RUNNING to CONCLUDED | Distribution of experiment durations |
| **False discovery estimate** | Expected false discoveries given current alpha and experiment volume | N_experiments × alpha (Bonferroni bound) or e-LOND estimate (ADR-018) |
| **Power distribution** | Histogram of statistical power across running experiments | Per-experiment power computed from current sample size and observed variance |

### Component 2: Learning Classification

At experiment conclusion, M5 prompts the experimenter to classify the outcome:

```protobuf
enum ExperimentLearning {
  EXPERIMENT_LEARNING_UNSPECIFIED = 0;
  EXPERIMENT_LEARNING_WIN = 1;           // Shipped positive treatment
  EXPERIMENT_LEARNING_REVERT = 2;        // Detected regression, reverted
  EXPERIMENT_LEARNING_NEUTRAL = 3;       // Confirmed no effect (valuable information)
  EXPERIMENT_LEARNING_PIVOT = 4;         // Inconclusive but informed new hypothesis
  EXPERIMENT_LEARNING_NO_LEARNING = 5;   // No actionable information gained
}

// Added to Experiment message:
ExperimentLearning learning_classification = 27;
string learning_notes = 28;  // free-text explanation
```

This enables the EwL learning rate calculation. M6 renders learning rate trends over time and by team/domain.

### Component 3: Optimal Alpha Recommendation

Following Netflix EC '25, M5 computes the portfolio-optimal significance threshold based on:
- Number of concurrent experiments
- Typical effect sizes observed in historical experiments
- Implementation costs (configurable per experiment type; near-zero for recommendation algorithm changes)
- User population size and traffic allocation

The recommendation engine in M5:

```go
type AlphaRecommendation struct {
    RecommendedAlpha float64  // often 0.10-0.20 for low-cost treatments
    Rationale        string
    ExpectedAnnualImpact float64  // at recommended vs. traditional alpha
    FalseDiscoveryRate   float64  // expected FDR at recommended alpha
}

func RecommendAlpha(
    historicalEffectSizes []float64,
    implementationCost float64,
    concurrentExperiments int,
    populationSize int64,
) AlphaRecommendation
```

This is advisory — M5 *recommends* an alpha during experiment creation but does not override the experimenter's choice. The recommendation appears in M6's create experiment form with an explanation.

### Component 4: Annualized Impact Projection

For shipped experiments, project the annualized business impact:

```protobuf
message AnnualizedImpact {
  string experiment_id = 1;
  // Observed treatment effect from the experiment.
  double observed_effect = 2;
  // Annualized projection using surrogate index approach.
  double annualized_effect = 3;
  double annualized_ci_lower = 4;
  double annualized_ci_upper = 5;
  // Method used for annualization.
  AnnualizationMethod method = 6;
  // Currency-denominated impact (if revenue metric).
  double annualized_revenue_impact = 7;
  string currency = 8;
}

enum AnnualizationMethod {
  ANNUALIZATION_LINEAR_EXTRAPOLATION = 0;  // simple: effect × 365/duration
  ANNUALIZATION_SURROGATE_INDEX = 1;       // Netflix approach: calibrated projection
  ANNUALIZATION_ORL = 2;                   // MDP-based (ADR-017)
}
```

M4a computes the annualized projection; M6 displays it in the portfolio dashboard as cumulative impact across all shipped experiments.

### Component 5: Traffic Allocation Optimizer

M5 provides a traffic allocation recommendation for concurrent experiments based on their priority and power requirements:

```go
type TrafficAllocation struct {
    ExperimentID string
    CurrentAllocation float64  // current % of layer
    RecommendedAllocation float64
    Rationale string
    // True if this experiment is underpowered at current allocation.
    Underpowered bool
    // Estimated days to significance at current vs. recommended allocation.
    DaysToSigCurrent float64
    DaysToSigRecommended float64
}

func OptimizeTrafficAllocation(
    runningExperiments []ExperimentState,
    layerCapacity map[string]int,  // buckets per layer
    priorities map[string]int,     // business priority 1-5
) []TrafficAllocation
```

Higher-priority experiments receive more traffic. Experiments near significance get a temporary traffic boost to reach conclusion faster. Experiments that are hopelessly underpowered (would need 6+ months at any allocation) are flagged for early termination review.

### Component 6: Decision Rule Evaluation

Following Netflix KDD '25, evaluate whether the platform's decision rules (alpha threshold, primary metric choice, CUPED usage) are calibrated to maximize business value:

1. Take historical concluded experiments (last 6 months).
2. Split into training (calibration) and test (evaluation) sets.
3. On training set: find the decision rule (alpha, metric, analysis method) that maximizes annualized shipped impact.
4. On test set: verify the rule produces similar results (avoid winner's curse).
5. Report the recommended decision rule with confidence intervals.

This runs as a periodic batch job (monthly) in M3, with results surfaced in the portfolio dashboard.

## Consequences

### Positive

- Transforms Kaizen from an experiment-by-experiment tool to a strategic experimentation program management platform.
- Learning classification (EwL) captures the value of experiments that find *nothing* — currently invisible in win-rate-only reporting.
- Optimal alpha recommendation could significantly increase the total value generated by the experimentation program, particularly for low-cost treatment changes (recommendation algorithms) where the traditional 0.05 threshold is unnecessarily conservative.
- Annualized impact projection gives leadership a single number: "the experimentation program generated $X in annualized revenue impact this quarter."
- Traffic allocation optimizer prevents the common failure mode of experiments running for months because they were allocated too little traffic.

### Negative

- Learning classification requires human input at experiment conclusion — an additional step that experimenters may skip. Mitigation: make classification mandatory for CONCLUDED → ARCHIVED transition; provide sensible defaults based on statistical results.
- Optimal alpha is politically sensitive. A recommendation of α = 0.15 may be perceived as "lowering the bar." The platform must communicate this as "optimizing total program value" not "accepting more false positives."
- Annualized impact projection requires strong assumptions about temporal stability (the effect observed in weeks 2-4 will persist for 12 months). These assumptions should be validated with the ORL framework (ADR-017) for high-impact experiments.
- Decision rule evaluation requires 6+ months of historical experiment data to be meaningful. New deployments of Kaizen will not benefit from this component initially.

### Risks

- Portfolio optimization may conflict with team-level autonomy. If the optimizer recommends reducing Team A's experiment traffic allocation to give more to Team B's higher-priority experiment, political tensions arise. Mitigation: recommendations are advisory; final allocation decisions remain with experiment owners and leadership.
- Annualized impact projection creates accountability expectations. If Q1's projected impact doesn't materialize in Q4's actual metrics, trust in the experimentation program suffers. Mitigation: always report projections with confidence intervals and validation against historical projection accuracy.

## Alternatives Considered

| Alternative | Pros | Cons | Why rejected |
|-------------|------|------|--------------|
| Per-experiment optimization only (status quo) | Simple; no coordination needed | Misses portfolio-level value; no aggregate impact tracking; no traffic optimization | Leaves significant business value on the table |
| Manual portfolio management (spreadsheets) | Flexible | Doesn't scale; no statistical foundation; prone to political allocation | The problem this ADR automates |
| Bayesian optimization of the portfolio (Meta Ax 1.0) | Theoretically optimal | Treats the experimentation program as a black-box optimization problem, which is inappropriate — experiments are not hyperparameters | Wrong abstraction level |
